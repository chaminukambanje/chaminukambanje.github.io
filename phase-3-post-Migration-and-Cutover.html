The data is in the cloud, but the system is read-only. This phase involves mapping users, verifying financials, and flipping the switch to "Live".

Step 1: User & Permission Mapping

On-premises SQL users (Windows Authentication) do not automatically translate to Microsoft Entra ID (Azure AD) users.

  1. Define User Mappings
    • Task: Open the "Cloud Migration Management" page.
    • Action: Use the "Define User Mappings" action to link legacy Windows logins to valid Microsoft 365 email addresses.
    • Note: Unmapped users will lose their permission sets upon cutover.
  2. Assign Licenses
    • Task: Ensure all users have a "Dynamics 365 Business Central" license assigned in the Microsoft 365 Admin Center.

Step 2: Disabling Migration & Going Live

Warning: This is the point of no return. Once migration is disabled, you cannot sync changes from the on-premises SQL server again without restarting.

  1. Final Delta Sync
    • Task: Run "Run Migration Now" one last time to catch any transactions posted while you were configuring users.
  2. Disable Cloud Migration
    • Action: Click "Disable Cloud Migration" in the setup page.
    • Result: This breaks the link to the SQL server, converts the tenant from "Read-Only" to "Read-Write", and finalizes the upgrade.
  3. Initialize Companies
    • Task: Open each company to trigger the final background initialization jobs.

Step 3: Validation & Clean-up

  • Financial Reconciliation: Run the "Trial Balance" report in both systems. If they differ by even 0.01, do not release the system to users.
  • Enable Job Queues: Scheduled tasks are paused by default during migration. You must manually set them to "Ready".
  • Update External Integrations: Repoint any Power BI dashboards, API connectors, or e-commerce sites to the new OData v4 web service URLs.
